# Design System Spacing

This document explains how to use our custom design system spacing tokens alongside Tailwind CSS's default spacing scale.

## Overview

Our design system provides semantic spacing tokens that map to specific pixel values while maintaining compatibility with Tailwind's default spacing system. This allows you to use both approaches in the same project.

## Available Spacing Tokens

| Token | Value | Equivalent Tailwind | Use Case |
|-------|-------|-------------------|----------|
| `xxs` | 8px (0.5rem) | `p-2` | Very tight spacing |
| `xs` | 12px (0.75rem) | `p-3` | Tight spacing |
| `sm` | 16px (1rem) | `p-4` | Small spacing |
| `m` | 20px (1.25rem) | `p-5` | Medium spacing |
| `md` | 24px (1.5rem) | `p-6` | Medium-large spacing |
| `lg` | 32px (2rem) | `p-8` | Large spacing |
| `xl` | 40px (2.5rem) | `p-10` | Extra large spacing |
| `xxl` | 48px (3rem) | `p-12` | Very large spacing |
| `xxxl` | 64px (4rem) | `p-16` | Maximum spacing |

## Usage Examples

### Padding

<CodeBlock language="tsx">
{`// Design system tokens
<div className="p-xxs">8px padding</div>
<div className="p-sm">16px padding</div>
<div className="p-lg">32px padding</div>

// Tailwind defaults (still work)
<div className="p-2">8px padding</div>
<div className="p-4">16px padding</div>
<div className="p-8">32px padding</div>`}
</CodeBlock>

### Directional Padding

<CodeBlock language="tsx">
{`// All directional variants work
<div className="px-sm py-lg">16px horizontal, 32px vertical</div>
<div className="pt-md pb-xs">24px top, 12px bottom</div>
<div className="pl-xl pr-m">40px left, 20px right</div>`}
</CodeBlock>

### Margin

<CodeBlock language="tsx">
{`// Margin tokens work the same way
<div className="m-sm">16px margin</div>
<div className="mx-lg my-md">32px horizontal, 24px vertical margin</div>
<div className="mt-xl mb-xs">40px top, 12px bottom margin</div>`}
</CodeBlock>

### Gap

<CodeBlock language="tsx">
{`// Gap utilities
<div className="flex gap-sm">16px gap between flex items</div>
<div className="grid gap-md">24px gap between grid items</div>`}
</CodeBlock>

## Mixed Usage

You can freely mix design system tokens with Tailwind's default spacing:

<CodeBlock language="tsx">
{`<div className="p-lg bg-white rounded-lg shadow-sm">
  <h2 className="text-xl font-semibold mb-sm">Card Title</h2>
  <p className="text-gray-600 mb-m">
    This card uses design system tokens for consistent spacing.
  </p>
  <div className="flex gap-4">
    <button className="px-4 py-2 bg-blue-500 text-white rounded">
      Tailwind spacing
    </button>
    <button className="px-md py-sm bg-purple-500 text-white rounded">
      Design system spacing
    </button>
  </div>
</div>`}
</CodeBlock>

## Implementation Details

The spacing tokens are defined in `src/app/globals.css` using Tailwind v4's `@theme inline` syntax:

<CodeBlock language="css">
{`@theme inline {
  /* Design System Spacing Tokens */
  --spacing-xxs: 0.5rem;   /* 8px - equivalent to p-2 */
  --spacing-xs: 0.75rem;   /* 12px - equivalent to p-3 */
  --spacing-sm: 1rem;      /* 16px - equivalent to p-4 */
  --spacing-m: 1.25rem;    /* 20px - equivalent to p-5 */
  --spacing-md: 1.5rem;    /* 24px - equivalent to p-6 */
  --spacing-lg: 2rem;      /* 32px - equivalent to p-8 */
  --spacing-xl: 2.5rem;    /* 40px - equivalent to p-10 */
  --spacing-xxl: 3rem;     /* 48px - equivalent to p-12 */
  --spacing-xxxl: 4rem;    /* 64px - equivalent to p-16 */
}`}
</CodeBlock>

## Benefits

1. **Semantic Naming**: Tokens like `sm`, `md`, `lg` are more meaningful than numeric values
2. **Consistency**: Ensures consistent spacing across your design system
3. **Flexibility**: Can still use Tailwind's default spacing when needed
4. **Maintainability**: Easy to update spacing values globally by changing CSS variables
5. **Type Safety**: Works with Tailwind's IntelliSense and autocomplete

## Best Practices

1. **Use design system tokens** for consistent spacing in your components
2. **Use Tailwind defaults** for one-off spacing needs or when working with third-party components
3. **Be consistent** within a component - don't mix approaches unnecessarily
4. **Document your choices** when deviating from the design system
